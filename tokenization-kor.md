# 토크나이제이션: 텍스트를 처리 가능한 단위로 나누기

토크나이제이션은 원시 텍스트를 기계학습 모델이 처리할 수 있는 개별 단위(토큰)로 변환하는 기초 과정입니다. 자연어 처리와 딥러닝의 교차점에 위치하며, 모델이 언어를 어떻게 인식하고 이해하는지 결정합니다.

## 원시 텍스트의 문제점

[신경망](/neural-kor.html)은 원시 텍스트 문자열을 직접 처리할 수 없습니다. 숫자 표현이 필요합니다. 텍스트를 임베딩하거나 변환하기 전에 먼저 어떤 것이 의미 있는 단위인지 결정해야 합니다. 단어일까요? 문자일까요? 부분단어일까요? 이 결정은 모델 성능, 어휘 크기, 다국어 능력에 연쇄 효과를 미칩니다.

## 단어 수준 토크나이제이션

가장 간단한 접근 방식은 공백과 구두점에서 텍스트를 분할합니다:

```
"안녕하세요, 세상!" → ["안녕하세요", ",", "세상", "!"]
```

**장점:**
- 직관적이고 사람이 읽기 쉬움
- 단어의 의미론적 의미 보존
- 잘 형식화된 영어 텍스트에서 작동

**단점:**
- 큰 어휘 크기 (영어 기준 100,000+ 토큰)
- 어휘 외(OOV) 단어를 처리할 수 없음
- 형태 변형을 별도 토큰으로 취급 ("running", "run", "runs")
- 명확한 단어 경계가 없는 언어에 부적합 (중국어, 일본어)

## 문자 수준 토크나이제이션

이 접근 방식은 모든 문자를 토큰으로 취급합니다:

```
"안녕" → ["안", "녕"]
```

**장점:**
- 최소 어휘 크기 (일반적으로 100-200 토큰)
- OOV 문제 없이 모든 텍스트 표현 가능
- 언어 무관적

**단점:**
- 매우 긴 시퀀스로 모델 용량에 부담
- 단어의 의미적 구조 손실
- 메모리 사용 비효율적
- 문자 수준에서 의미 있는 표현 학습 어려움

## 부분단어 토크나이제이션: 현대적 접근법

부분단어 토크나이제이션은 단어와 문자 수준 사이의 균형을 찾아, 단어를 의미 있는 부분단어 단위로 분할합니다. 세 가지 지배적 알고리즘이 등장했습니다:

### 바이트 쌍 부호화 (Byte Pair Encoding, BPE)

BPE는 가장 빈번한 인접 문자/토큰 쌍을 반복적으로 병합합니다:

```
초기: "안녕하세요" → ["안", "녕", "하", "세", "요"]
가장 빈번한 쌍 병합: ["안녕", "하", "세", "요"]
다음 빈번한 쌍 병합: ["안녕", "하세", "요"]
최종: ["안녕하세요"]
```

[GPT](/gpt-kor.html) 모델에서 사용됩니다. 적절한 시퀀스 길이를 유지하면서 컴팩트한 어휘를 생성합니다 (일반적으로 50K 토큰).

### 워드피스 (WordPiece)

구글의 WordPiece 알고리즘은 사전에 존재하는 가장 긴 토큰을 탐욕적으로 구성합니다:

```
"놀고있다" → ["놀", "##고", "##있다"]
"재시드" → ["재", "##시", "##드"]
```

"##" 접두사는 부분단어 계속을 나타냅니다. BERT에서 사용되므로, [BERT](/bert-kor.html) 어휘에는 많은 "##" 토큰이 포함됩니다. 어휘 크기는 일반적으로 30K 토큰입니다.

### 유니그램 언어 모델 (Unigram Language Model)

이 확률적 접근 방식은 토크나이제이션을 최대 우도 문제로 취급합니다:

```
P("안녕하세요" → ["안", "녕", "하", "세", "요"]) vs P("안녕하세요" → ["안녕", "하세요"])
```

유니그램 언어 모델 하에서 전체 확률을 최대화하는 토크나이제이션을 선택합니다.

### 센텐스피스 (SentencePiece)

바이트에서 직접 작동하는 언어 무관적 토크나이제이션 방법으로, 유니코드, 공백, 특수 문자를 원활하게 처리합니다:

```
"안녕 세상" → ["▁안녕", "▁세상"]
"こんにちは" (일본어) → ["▁", "こ", "ん", "に", "ちは"]
```

T5, mBART, XLNet에서 사용됩니다. 다국어 모델에 특히 효과적입니다.

## 어휘 크기 트레이드오프

어휘 크기 선택은 여러 요소 간 균형이 필요합니다:

| 어휘 크기 | 영향 |
|---|---|
| **작음 (10K)** | 더 짧은 시퀀스, 작은 임베딩, 하지만 빈번한 OOV와 정보 손실 |
| **중간 (30-50K)** | 대부분 모델의 최적점; 합리적인 시퀀스 길이와 메모리 사용 |
| **큼 (100K+)** | 더 긴 시퀀스, 큰 임베딩, 희귀 단어의 더 나은 보존이지만 메모리 오버헤드 |

## 다국어 토크나이제이션 도전

언어마다 최적의 토크나이제이션 전략이 크게 다릅니다:

**형태론적으로 복잡한 언어** (터키어, 핀란드어, 헝가리어): 복잡한 접사 처리를 위해 부분단어 토크나이제이션 필요

**명시적 단어 경계 없음** (중국어, 일본어): 문자 기반 또는 SentencePiece 접근법이 종종 우수

**혼합 스크립트** (음차, 코드믹싱): 표준 토크나이저 실패; SentencePiece의 바이트 수준 접근법이 더 잘 처리

다국어 모델은 일반적으로 통합 어휘를 사용합니다 (예: mBERT의 119K 어휘는 104개 언어를 포함).

## 토크나이제이션과 모델 성능

토크나이제이션 선택은 하위 성능에 영향을 미칩니다:

1. **정보 손실**: 공격적인 부분단어 토크나이제이션은 형태론적 신호를 잃을 수 있음
2. **시퀀스 길이**: 더 긴 토큰화된 시퀀스는 더 많은 메모리와 계산 소비
3. **임베딩 품질**: 단어 수준 어휘는 고품질 임베딩 가능; 부분단어 어휘는 의미를 분할
4. **미세조정**: 일부 토크나이제이션 선택은 특정 하위 작업에 도움
5. **희귀 단어**: 부분단어 토크나이제이션은 도메인 특화 용어 처리 향상 (의료, 기술)

연구는 최적 어휘 크기가 언어와 작업에 따라 다름을 보여줍니다: 영어는 30-50K, 일본어는 50-100K, 형태론적으로 복잡한 언어는 더 공격적인 부분단어 분할을 가진 더 작은 어휘 (16-32K)에서 이점이 있습니다.

## 현대적 토크나이제이션 전략

최신 모델은 하이브리드 접근법을 사용합니다:

- **접두사 토크나이제이션**: 특수 토큰과 프롬프트 구조의 별도 처리 (명령 미세조정 모델에서 중요)
- **앵커 토크나이제이션**: 특정 기능을 위한 예약 토큰 (추론 토큰, 검색 토큰)
- **어휘 확장**: 미세조정 중 동적 도메인 특화 토큰 추가

## 실제 적용

NLP 시스템을 구축할 때:

1. 토크나이저를 모델과 일치: 사전 학습 모델과 함께 출시된 정확한 토크나이저 사용
2. OOV 비율 모니터링: 높은 OOV는 어휘 불일치를 암시
3. 하위 작업 고려: 번역은 형태론적 인식에서 이점; 분류는 공격적인 부분단어 분할 허용 가능
4. 실제 데이터로 평가: 토크나이제이션 성능은 도메인 간 극적으로 변함
5. 특수 토큰 처리: `[CLS]`, `[SEP]`, `[PAD]` 등의 적절한 처리는 중요

## 결론

토크나이제이션은 단순해 보이지만 하위의 모든 것을 결정합니다. 단어 수준에서 부분단어 토크나이제이션으로의 전환은 현대 NLP 성공에 중요했으며, 형태론적 복잡성, 희귀 단어, 다국어 컨텍스트 처리를 가능하게 했습니다. 토크나이제이션 전략을 이해하면 특정 모델이 특정 작업에서 우수한 이유를 설명하고 새로운 응용을 위한 모델 선택 및 아키텍처 설계 결정에 정보를 제공합니다.
