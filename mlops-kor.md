# MLOps와 모델 배포

## 소개

Machine Learning Operations(MLOps)는 모델 개발과 프로덕션 배포 사이의 간격을 메웁니다. 데이터 수집부터 모니터링 및 재교육까지 전체 ML 생명주기를 효율적으로 관리하기 위한 실습, 도구, 문화를 포함합니다. MLOps는 신뢰할 수 있는 확장 가능한 AI 시스템 구축에 필수적입니다.

## ML 생명주기

MLOps는 여러 상호 연결된 단계를 관리합니다:

**데이터 수집 및 준비**: 원시 데이터 수집, 정제, 검증 및 버전 관리. 고품질 데이터는 기초이며, 입력이 나쁘면 출력도 나쁩니다.

**특성 엔지니어링**: 원시 데이터를 의미 있는 특성으로 변환합니다. 종종 도메인 지식과 반복적 실험이 필요합니다.

**모델 개발**: 학습 및 평가. 데이터 과학자들은 아키텍처, 하이퍼파라미터, 알고리즘으로 실험합니다. 이 단계는 수십 개의 모델 후보를 생성합니다.

**모델 검증**: 보류 테스트 세트에 대한 엄격한 평가. 교차 검증, 계층화, 시간적 검증은 일반화를 보장합니다.

**배포**: 모델을 프로덕션으로 이동합니다. 컨테이너화, API 배포, 점진적 롤아웃은 신뢰성을 보장합니다.

**모니터링 및 재교육**: 모델 성능의 지속적 모니터링, 드리프트 감지, 재교육 트리거. 모델은 데이터 분포가 변함에 따라 시간이 지남에 따라 성능이 저하됩니다.

## ML을 위한 CI/CD

전통적 CI/CD 원칙은 ML에 적용되지만 중요한 수정이 필요합니다:

**ML을 위한 지속적 통합**:
- 코드 변경의 자동화된 테스트
- 데이터 검증 및 스키마 확인
- 모델 품질 게이트 (최소 성능 임계값)
- 재현성 확인 (고정 시드, 문서화된 종속성)

**ML을 위한 지속적 배포**:
- 프로덕션 릴리스 전 자동화된 모델 검증
- 카나리 배포로 점진적 롤아웃
- 새로운 모델을 위한 A/B 테스트 프레임워크
- 성능 저하 시 쉬운 롤백 메커니즘

소프트웨어 CI/CD와의 주요 차이: ML 시스템은 비결정론적입니다. 동일한 데이터와 동일한 하이퍼파라미터로 학습한 두 모델은 임의의 초기화로 인해 약간 다를 수 있습니다.

## 실험 추적

### MLflow

MLflow는 ML 생명주기 관리를 위한 오픈 소스 플랫폼입니다:

```
mlflow.log_param("learning_rate", 0.001)
mlflow.log_metric("accuracy", 0.95)
mlflow.log_model(model, "model")
```

기능:
- 파라미터, 메트릭, 아티팩트 로깅
- 모델 계보 및 버전 추적
- 모델 레지스트리 및 배포 도구 제공
- 대부분 ML 프레임워크와 통합

### Weights & Biases (W&B)

W&B는 더 풍부한 시각화 및 협업을 제공합니다:
- 스윕(Sweeps): 자동화된 하이퍼파라미터 최적화
- 보고서: 공유 가능한 실험 문서
- 아티팩트: 데이터셋, 모델, 출력 추적
- 알림: 프로덕션 모델 메트릭 모니터링

## 모델 레지스트리

모델 레지스트리는 프로덕션 모델의 단일 정보 소스로 작동합니다:

**핵심 기능**:
- 버전 제어: 여러 모델 버전 추적
- 메타데이터 저장소: 파라미터, 학습 날짜, 성능 메트릭
- 단계 관리: 스테이징, 프로덕션 환경을 통한 모델 전환
- 계보 추적: 어느 데이터와 코드가 어느 모델을 생성했는지 기록

**인기 솔루션**:
- MLflow 모델 레지스트리: 간단함, MLflow와 통합
- Hugging Face 모델 허브: 커뮤니티 주도, 트랜스포머에 좋음
- 맞춤형 솔루션: AWS SageMaker, Google Vertex AI 레지스트리

## 프로덕션의 A/B 테스트

A/B 테스트는 실제 사용자 트래픽에서 두 모델 버전을 비교합니다:

**설정**:
- 트래픽의 비율을 대조(현재 모델)와 치료(새로운 모델)로 라우팅
- 두 그룹 모두에서 메트릭 수집: 정확도, 지연 시간, 사용자 만족도
- 통계적 유의성을 위해 실행 (일반적으로 2-4주)

**도전 과제**:
- 샘플 크기: 통계적 검정력을 위해 충분한 트래픽 필요
- 상호 작용 효과: 새로운 모델이 다운스트림 시스템에 영향을 미칠 수 있음
- 식인 현상: 두 버전 모두 동일한 데이터 소스에서 학습할 수 있음
- 다중 테스트: 많은 모델 테스트는 거짓 양성률 증가

**대안**:
- 인터리빙(Interleaving): A/B보다 효율적, 요청 당 모델 교대
- 문맥적 밴디트: 적응형 할당, 더 나은 성능 모델로 더 많은 트래픽

## 모니터링 및 드리프트 감지

### 데이터 드리프트

시간 경과에 따른 입력 특성의 분포 이동:
- 공변량 드리프트: P(X)가 변하지만 P(Y|X)는 같음
- 레이블 드리프트: P(Y)가 변하지만 P(X|Y)는 같음
- 개념 드리프트: P(Y|X)가 변함 (가장 문제가 됨)

**감지 방법**:
- 통계 테스트: Kolmogorov-Smirnov, 카이제곱
- 거리 메트릭: Wasserstein, Maximum Mean Discrepancy
- 재구성 오류: 기준선에 오토인코더 학습, 새 데이터에 대한 오류 확인

### 모델 성능 모니터링

- 프로덕션에서 주요 메트릭 추적 (정확도, 정밀도, 재현율)
- 비정상 성능 저하에 대한 경고 임계값 설정
- 학습 세트 성능과 기준선으로 비교
- 자원 제약을 위한 지연 시간 및 처리량 모니터링
- 오프라인 분석을 위한 예측 및 실제 레이블 로깅

### 재교육 전략

**트리거 기반**: 성능이 임계값 아래로 떨어지면 재교육

**스케줄 기반**: 성능에 관계없이 주간/월간 재교육

**데이터 기반**: 데이터 드리프트 모니터링, 감지될 때 재교육

**하이브리드**: 견고성을 위해 접근 방식 결합

## 모델 서빙 아키텍처

**배치 예측**: 높은 지연 시간, 높은 처리량 (오프라인 분석용)

**온라인 서빙**: 낮은 지연 시간 요구 (일반적으로 100ms 미만)

**스트리밍**: 연속 데이터 스트림에서 실시간 예측

**엣지 추론**: 모바일/IoT의 온디바이스 예측

## 결론

MLOps는 모델 개발을 연구 운동에서 프로덕션 엔지니어링으로 변환합니다. 성공은 도구, 데이터 인프라, 모니터링, 조직 프로세스에 대한 투자가 필요합니다. MLOps 없이 최고의 모델이 실패하고, 성숙한 운영으로 평범한 모델이 성공합니다.
