# KV 캐시와 추론 최적화

자동회귀 생성 중, 언어 모델은 토큰을 하나씩 예측합니다. 최적화 없이 이 과정은 이전 토큰에 대한 어텐션을 반복적으로 재계산하여 계산을 낭비합니다. KV 캐시(키-밸류 캐시)는 사전 계산된 키와 밸류를 캐싱하고 재사용하여 추론을 가속화하는 중요한 최적화입니다.

## 자동회귀 병목현상

자동회귀 디코딩은 단계별로 시퀀스를 생성합니다:

```
단계 1: 입력 "Hello"     → 출력 "world"
단계 2: 입력 "Hello world" → 출력 "is"
단계 3: 입력 "Hello world is" → 출력 "beautiful"
```

근본적 문제: 단계 t에서 모델은 토큰 t+1을 예측하기 위해 모든 t개 토큰을 다시 처리합니다. 토큰 1..t-1에 대한 어텐션은 동일하게 남아 있음에도 불구하고.

길이 n인 위치 시퀀스 예시:
```
단계 1: [token_1]에 대한 어텐션 계산                    (1개 토큰)
단계 2: [token_1, token_2]에 대한 어텐션 계산           (2개 토큰)
단계 3: [token_1, token_2, token_3]에 대한 어텐션 계산  (3개 토큰)
...
단계 n: [token_1, ..., token_n]에 대한 어텐션 계산      (n개 토큰)

총 연산: 1 + 2 + 3 + ... + n = n(n+1)/2 = O(n²)
```

이 이차 복잡도는 긴 시퀀스에 대한 생성을 극도로 느리게 만듭니다.

## KV 캐시 메커니즘

키와 밸류를 재계산하는 대신, KV 캐시는 이들을 저장합니다:

```
# 초기 순전파 (프리필):
inputs = [token_1, token_2, ..., token_n]
Q_new, K_all, V_all = compute(inputs)
cache = {K: K_all, V: V_all}

# 단계 n+1 (디코드):
new_token = token_n+1
Q_new, K_new, V_new = compute([new_token])
K_cached = cache.K + K_new          # 연결
V_cached = cache.V + V_new
attention_output = softmax(Q_new @ K_cached.T) @ V_cached
cache.update({K: K_cached, V: V_cached})
```

핵심 통찰: 새 토큰의 Q만 계산되고, 이전 토큰의 K와 V는 재사용됩니다.

## 메모리와 계산 트레이드오프

KV 캐시는 메모리를 계산 속도로 교환합니다:

**캐시 없음 (순진):**
- 계산: n-길이 시퀀스에 대해 O(n²)
- 메모리: 순전파 당 O(n) (지속성 없음)

**KV 캐시 포함 (최적화):**
- 계산: n-길이 시퀀스에 대해 O(n) (선형!)
- 메모리: O(n * d_model) 지속 저장소

13B 매개변수 모델, 시퀀스 길이 2048:
- K 캐시: 2048 × 5120 차원 × 2 바이트 (fp16) ≈ 헤드당 20 MB
- V 캐시: 2048 × 5120 차원 × 2 바이트 ≈ 헤드당 20 MB
- 40개 헤드 전체: 시퀀스당 ~1.6 GB

이는 추론 중 배치 크기가 제한되는 이유를 설명합니다—각 시퀀스는 자신의 캐시가 필요합니다.

## 멀티배치 추론과 메모리 관리

여러 시퀀스에 대한 KV 캐시 관리는 중요합니다:

```python
# 배치 생성
batch_size = 32
sequence_length = 2048
num_heads = 40
head_dim = 128

kv_cache_size = batch_size * sequence_length * num_heads * head_dim * 2 bytes
              = 32 * 2048 * 40 * 128 * 2
              ≈ 52 GB (fp16)
```

현대적 최적화 전략:

1. **배치별 할당**: 예상 배치 크기에 대해 캐시 사전 할당
2. **동적 할당**: 시퀀스 길이 증가에 따라 캐시 확장
3. **재사용 전략**: 새 요청을 위해 가장 오래된 캐시 항목 제거
4. **양자화**: 캐시를 int8 또는 fp8로 저장 (감소된 정밀도)

## PagedAttention (vLLM)

vLLM에 도입된 PagedAttention은 메모리 효율성을 극적으로 개선합니다:

**기존 접근법:**
- 시퀀스별로 연속적으로 할당된 KV 캐시
- 단편화: 시퀀스가 일찍 끝나면 부분 블록 낭비
- 비효율적: 시퀀스 간 캐시 공유 불가

**PagedAttention 접근법:**
- KV 캐시를 고정 "페이지"로 나눔 (예: 페이지당 16개 토큰)
- 페이지는 가상 (물리 메모리로 매핑됨)
- 시퀀스는 물리 페이지 공유 (copy-on-write)
- OS 가상 메모리 같은 동적 메모리 할당

### 2-토큰 페이지 예시:
```
시퀀스 1: 토큰 [0,1] [2,3] [4,5] ...
시퀀스 2: 토큰 [0,1] [2,3] [4,5] ...

두 시퀀스가 동일 접두사 [0,1][2,3]을 사용하면,
동일 물리 페이지를 공유할 수 있습니다!
```

이점:
- **메모리 효율성**: 5-10배 개선 (60-75% GPU 메모리 절감)
- **더 높은 처리량**: 동일 GPU 메모리로 더 많은 시퀀스 배치
- **감소된 지연시간**: 더 빠른 토큰 생성
- **접두사 공유**: 공통 접두사에 대한 계산 재사용

## 추측적 디코딩

추측적 디코딩은 드래프트를 사용하여 추론을 가속합니다:

```
# 드래프트 모델이 k개 토큰을 빠르게 생성
draft_tokens = draft_model.generate(k=5)

# 메인 모델이 모든 k개 토큰을 병렬로 검증
verifications = main_model.verify(draft_tokens, batch=True)

# 각 토큰 수락 또는 거부
for i, verified in enumerate(verifications):
    if verified:
        accept draft_tokens[i]
    else:
        continue with main model output
```

핵심 특성:
- 더 작은/빠른 "드래프트" 모델을 사용하여 후보 생성
- 메인 모델이 배치 처리를 사용하여 병렬로 검증
- 품질 손실 없음: 최종 출력은 메인 모델에서 나옴
- 속도 향상: 드래프트 정확도가 높을 때 2-3배

일반적 드래프트 모델:
- 동일 아키텍처, 더 작은 크기 (예: 70B에 대해 7B)
- 메인 모델 출력에서 학습된 증류 버전
- 더 저렴한 양자화 변형

## 지속적 배칭

지속적 배칭(동적 배칭)은 처리량을 개선합니다:

**정적 배칭 (기존):**
- 모든 시퀀스가 현재 반복을 완료할 때까지 대기
- 가장 느린 시퀀스를 대기하는 동안 GPU 유휴
- 가변 길이 시퀀스에 비효율적

**지속적 배칭 (스트리밍):**
- 한 시퀀스가 완료되자마자 제거하고 새 시퀀스 추가
- 가장 느린 시퀀스를 대기하지 않음
- GPU는 높은 사용률 유지

구현 전략:
```
대기 중 요청 큐: [seq_1, seq_2, seq_3, ...]
반복 t에서 배치: [seq_1(t), seq_2(t), seq_3(t)]
반복 t+1에서 배치: [seq_2(t+1), seq_3(t+1), seq_4(t+1)]
                (seq_1 완료, seq_4 추가)
```

처리량 개선: 정적 배칭 대비 2-4배.

## 메모리 효율적 추론 기법

### 멀티 쿼리 어텐션 (MQA)
- 모든 쿼리 헤드에서 공유되는 하나의 키/밸류 헤드
- KV 캐시를 h배로 감소 (h = 헤드 수)
- 트레이드오프: 약간의 품질 손실, 막대한 메모리 절감

### 그룹 쿼리 어텐션 (GQA)
- 중간: Q 헤드보다 적은 KV 헤드 (예: 8 KV, 40 Q)
- MQA와 표준 MHA 사이 균형
- 현대 모델에서 인기 (LLaMA 2, Mistral)

### 양자화 KV 캐시
- 키/밸류를 int8 또는 더 낮은 정밀도로 저장
- 50% 메모리 감소 (fp16 → int8)
- 적절한 양자화로 최소 품질 손실
- 메모리를 다른 요소의 병목으로 감소

## 제약사항과 스케일링

KV 캐시 제약사항은 모델 배포를 결정합니다:

1. **메모리 한계**: 제한된 GPU 메모리가 최대 배치 크기 제한
2. **지연시간-처리량 트레이드오프**: 더 많은 배칭 → 더 높은 처리량이지만 더 나쁜 지연시간
3. **시퀀스 길이 제한**: 더 긴 시퀀스는 더 많은 캐시 필요
4. **Flash Attention 제한**: 빠른 커널도 캐시가 처리량 제한

현대 LLM 서비스는 다음을 통해 균형을 맞춥니다:
- 스마트 배칭 스케줄
- 요청 큐 관리
- 우선순위 스케줄링 (짧은 시퀀스 우선)
- 조기 종료 전략

## 결론

KV 캐시는 자동회귀 디코딩을 O(n²)에서 O(n)으로 변환하여 생성을 실용적으로 만듭니다. PagedAttention, 지속적 배칭, 추측적 디코딩 같은 혁신은 효율성을 더 밀어붙입니다. 캐시 관리 이해는 대규모 LLM 배포, 추론 비용 예측, 토큰 처리량 최적화에 필수적입니다.

순진한 것과 최적화된 추론 간의 차이는 처리량과 지연시간에서 수 배 차이일 수 있습니다—KV 캐시를 현대 LLM 배포의 가장 높은 영향 최적화 중 하나로 만듭니다.
