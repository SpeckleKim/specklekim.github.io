# AI 학습 인프라

## 소개

대형 AI 모델 학습은 특화된 인프라가 필요합니다: 분산 [GPU](/gpu-hardware-kor.html) 클러스터, 최적화된 네트워킹, 결함 내성, 모니터링. 현대 대형 모델([LLM](/llm-kor.html), [확산 모델](/diffusion-models-kor.html))은 수천 개의 GPU가 여러 데이터 센터에서 필요하며, 조정된 학습 및 효율적인 통신이 필요합니다. 인프라 이해는 규모 있는 모델을 학습하는 모든 사람에게 중요합니다.

## GPU 클러스터 아키텍처

### 단일 머신 설정

**일반적인 고성능 머신**:
- 8x A100 또는 H100 [GPU](/gpu-hardware-kor.html)
- [GPU](/gpu-hardware-kor.html) 간 NVLink 상호연결 (A100에서 방향당 425 GB/s)
- 2TB+ CPU RAM
- 100TB+ NVMe 저장소
- 이중 10GbE 네트워크 인터페이스

**사용 사례**: 모델 병렬성으로 50-100B 파라미터까지의 모델.

### 다중 머신 클러스터

**로컬 클러스터** (동일 데이터 센터):
- 10-1000 머신
- InfiniBand로 연결 (200-400 Gbps)
- 노드 간 서브 밀리초 지연 시간
- 페타바이트 규모 저장소 시스템

**분산 클러스터** (여러 데이터 센터):
- 지역 전체 머신
- 더 높은 지연 시간, 낮은 대역폭
- 신중한 동기화 필요
- 지역 컴퓨팅 가용성 처리

## 고속 상호연결

### NVLink

NVIDIA의 고대역폭 [GPU](/gpu-hardware-kor.html)-대-GPU 연결:
- A100: 400 GB/s (NVLink 4로 600 GB/s)
- H100: 900 GB/s
- 규모 있는 올-리듀스 작업 가능
- 네트워크 트래픽과 공유되지 않음

### InfiniBand

고속 네트워킹 패브릭:
- HDR InfiniBand: 방향당 200 Gbps
- NDR InfiniBand: 방향당 400 Gbps
- 서브 마이크로초 지연 시간
- RDMA (원격 직접 메모리 액세스) 지원

### 400GbE 이더넷

데이터 센터 네트워킹의 신흥 표준:
- 400 Gbps 전이중
- InfiniBand보다 낮은 비용
- 약간 높은 지연 시간
- 덜 대역폭 제약이 있는 애플리케이션에 좋음

## 분산 학습 프레임워크

### PyTorch 분산 데이터 병렬 (DDP)

[GPU](/gpu-hardware-kor.html) 전체에서 단순 데이터 병렬성:

```python
model = MyModel()
ddp_model = DistributedDataParallel(model)
# 장치 간 자동 경사 동기화
```

**작동 방식**:
1. 각 GPU에 모델 복사
2. 각 GPU는 다른 배치 처리
3. 로컬에서 경사 계산
4. 경사 동기화를 위해 올-리듀스
5. 모델 가중치 동일하게 업데이트

**제한**: 모델이 단일 [GPU](/gpu-hardware-kor.html) 메모리에 맞아야 함.

### 텐서 병렬성

모델 파라미터를 [GPU](/gpu-hardware-kor.html) 간에 분할:

```
계층 1: [param_split_1][param_split_2][param_split_3]
        GPU1            GPU2           GPU3
```

- 단일 GPU에 너무 큰 모델에 유용
- 세분화된 동기화 필요
- 절충: 통신 오버헤드 vs 모델 크기

### 파이프라인 병렬성

모델 계층을 [GPU](/gpu-hardware-kor.html) 간에 분할:

```
GPU1: [계층1, 계층2]
GPU2: [계층3, 계층4]
GPU3: [계층5, 계층6]
```

- 매우 깊은 모델에 좋음
- GPU는 순차적으로 다른 계층 처리
- 신중한 배칭 전략 필요 (GPipe, Megatron)
- [GPU](/gpu-hardware-kor.html) 유휴 시간 가능성

### Mixture of Experts (MoE)

다른 입력에 대해 다른 파라미터:

```
입력 -> 라우터 -> 전문가 1 -> 출력
              -> 전문가 2 ->
              -> 전문가 3 ->
```

- 희소 계산: 입력당 파라미터 서브셋 활성화
- 정비례 계산 없이 파라미터 수 확장
- Switch Transformers, Grok 모델에서 성공적으로 사용
- 학습 효율을 약간 감소 (전문가 불균형)

## 통신 최적화

### 경사 압축

동기화 중 통신 볼륨 감소:

**저순위 업데이트**: 중요 경사 업데이트만 통신

**상위 k 희소화**: 가장 큰 경사 값만 송신

**양자화**: 송신 전 경사의 정밀도 감소

**이점**: 최소 정확도 손실로 5-10배 통신 감소.

### 계산과 통신 겹침

통신 지연 시간 숨김:

```
N 계층의 경사가 통신되는 동안,
N+1 계층의 경사 계산
```

최신 프레임워크 (PyTorch, JAX)는 자동으로 겹침 지원.

### 링 올-리듀스

많은 노드 클러스터에 대한 효율적 집단 작업:

```
노드 0 -> 노드 1 -> 노드 2 -> ... -> 노드 N -> 노드 0
로컬 경사 세그먼트 송신, 수신 및 집계
```

- 스타 토폴로지보다 효율적
- 클러스터 크기에 따라 더 잘 확장

## 체크포인트 및 결함 내성

### 경사 체크포인트

계산을 위해 메모리 교환:

```
전방 패스: 체크포인트만 계산, 중간 활성화 버림
역방 패스: 체크포인트에서 중간 활성화 재계산
```

- 최소 속도 저하로 메모리 50% 감소
- 대형 모델에 필수

### 모델 체크포인트

주기적으로 모델 상태 저장:
- 각 에포크 또는 고정 간격
- 결함 시 체크포인트에서 재개 가능
- 디스크 공간이 병목이 될 수 있음

**전략**:
- 마지막 N개 체크포인트만 유지
- 초기에 빠른 저장소(NVMe)에 저장, 느린 아카이브로 이동
- 학습 방해를 피하기 위한 비동기 쓰기

### 비동기 체크포인트

학습이 계속되는 동안 저장:
- CPU 메모리에 쓰기, 백그라운드에서 디스크로 전송
- 학습 중단 감소
- 버퍼링을 위해 충분한 RAM 필요

### 분산 체크포인트

여러 머신에 걸쳐 모델 상태 저장:
- 저장소의 단일 실패 지점 피하기
- 병렬 쓰기로 더 빠른 저장/복원
- 조정된 복구 필요

## 비용 최적화

### 스팟 인스턴스

더 저렴한 선점 가능 VM (70-80% 할인):
- 위험: 인스턴스를 언제든지 회수 가능
- 완화: 빈번한 체크포인트
- 중요하지 않은 실험에 좋음

### 혼합 인스턴스 유형

다른 [GPU](/gpu-hardware-kor.html) 유형 사용:
- 덜 요구하는 작업에 이전 A100
- 최첨단 연구에 최신 H100
- 시간 당 비용 감소

### 배치 학습 스케줄

피크 시간 외에 학습:
- 일부 지역에서 30-50% 비용 절감
- 유연한 학습 스케줄 필요
- 시간 비판적이지 않은 작업에 좋음

### 자동 확장

동적으로 클러스터 크기 조정:
- 100 GPU로 시작, 피크 시 1000으로 확장
- 수요 감소 시 감소
- 인프라 지원 필요 (Kubernetes, Ray)

## 모니터링 및 디버깅

### GPU 사용률

실제 [GPU](/gpu-hardware-kor.html) 계산 추적:
- 이상적: 85-95% 사용률 (항상 계산)
- 낮은 사용률: 데이터 로딩 또는 동기화의 병목
- 도구: nvidia-smi, PyTorch 프로파일러

### 네트워크 대역폭

통신 패턴 모니터링:
- 올-리듀스가 병목? 통신 겹침 시도
- 불균일한 트래픽? 왜곡된 모델 병렬성 가능
- 도구: Netstat, 맞춤형 로깅

### 열 관리

GPU는 열을 발생:
- 온도 모니터링
- 과열 시 클록 속도 감소
- 적절한 냉각 인프라 보장
- 80-85°C에서 학습이 최적

### 학습 수렴

체크포인트 간 손실 곡선 추적:
- 이상은 버그 또는 하이퍼파라미터 문제 가능
- 다른 학습 실행 비교
- 실험 추적 사용 (MLflow, W&B)

## 인기있는 클라우드 플랫폼

**AWS**: A100/H100이 있는 EC2 인스턴스, 좋은 스팟 가격

**Google Cloud**: TPU 클러스터 및 [GPU](/gpu-hardware-kor.html) 옵션, JAX/TensorFlow와 통합

**Azure**: [GPU](/gpu-hardware-kor.html) 클러스터, PyTorch 생태계와 통합

**Lambda Labs**: [GPU](/gpu-hardware-kor.html) 렌탈, 간단한 인터페이스, 실험에 좋음

## 미래 방향

- **포토닉 상호연결**: 전기 InfiniBand를 광학으로 대체
- **고급 냉각**: 액체 냉각으로 더 높은 밀도 가능
- **소프트웨어 정의 네트워크**: 더 유연한 통신 패턴
- **이질적 계산**: 작업별로 최적화된 [GPU](/gpu-hardware-kor.html), TPU, CPU 혼합

## 결론

현대 AI 학습은 알고리즘처럼 심각하게 인프라를 생각해야 합니다. 규모 있는 성공적인 학습은 [분산 학습](/distributed-training-kor.html) 패턴, 통신 최적화, 결함 내성의 이해가 필요합니다. 모델이 커질수록 인프라는 가능성과 비용 효율성에 점점 더 중요해집니다.
