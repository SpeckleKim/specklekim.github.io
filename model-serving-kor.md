# 모델 서빙과 API 설계

## 소개

모델 서빙은 ML 파이프라인의 마지막 단계입니다: 학습된 모델을 프로덕션으로 가져가 새로운 데이터에 대한 예측을 만드는 것입니다. 모델 서빙 시스템은 경쟁하는 요구사항을 균형잡아야 합니다: 낮은 지연 시간, 높은 처리량, 신뢰성, 자원 효율성. 설계가 부실한 서빙 계층은 우수한 모델도 병목이 될 수 있습니다.

## 인기있는 모델 서빙 프레임워크

### TorchServe

PyTorch의 프로덕션 서빙 프레임워크:
- PyTorch 모델에서 HTTP API 생성
- 내장 배칭 및 [GPU](/gpu-hardware-kor.html) 지원
- 모델 관리: 버전 관리, A/B 테스트
- 확장 가능: 단일 서버 또는 분산 클러스터

```python
# Handler는 요청/응답 로직을 정의
def handle(data, context):
    model = context.model
    return model(data)
```

### TensorFlow Serving

Google의 프로덕션 ML 서빙:
- TensorFlow 모델에 최적화
- SavedModel 형식 지원
- 모델 버전 관리 및 카나리 배포 내장
- gRPC 및 REST API 엔드포인트

### vLLM

대형 언어 모델 추론 특화:
- 토큰 수준 배칭 (불완전한 시퀀스 배칭)
- 페이지 지정 주의 메커니즘 (메모리 효율)
- 다중 [GPU](/gpu-hardware-kor.html) 추론 지원
- LLM의 최고 처리량

### Triton Inference Server

NVIDIA의 추론 플랫폼:
- 여러 프레임워크 지원 (TensorFlow, PyTorch, ONNX)
- 설정 가능한 지연을 포함한 동적 배칭
- [GPU](/gpu-hardware-kor.html) 메모리 최적화
- 배포 용이를 위한 모델 저장소

### ONNX Runtime

최적화된 추론 런타임:
- 프레임워크 무관 (모델을 ONNX 형식으로 변환)
- 크로스 플랫폼 (CPU, [GPU](/gpu-hardware-kor.html), 모바일, 웹)
- 그래프 최적화 및 [양자화](/quantization-kor.html)
- 낮은 지연 시간 추론

## 배칭 전략

### 정적 배칭

고정 배치 크기, 배치가 가득 찰 때까지 대기:
- 예측 가능한 지연 시간 및 처리량
- 요청 비율이 변하면 낭비적
- 트래픽 패턴 이해 필요

### 동적 배칭

변수 배치 크기, 타임아웃 기반:
- T 밀리초까지 배치 대기
- B개 요청까지 축적
- 변수 로드 시 더 효율적
- 위험: 타임아웃이 다가오면 높은 지연 시간

**예시**:
- 최대 배치 크기: 32
- 최대 대기: 100ms
- 100ms 내에 64개 요청 도착: 32개씩 두 배치
- 16개 요청, 더 이상 도착 없음: 100ms 후 16개 배치 전송

### 토큰 수준 배칭 (LLM용)

LLM은 순차적으로 토큰을 생성합니다. vLLM은 도착시 요청을 [토큰화](/tokenization-kor.html), 토큰 배칭:
- 최대 길이 패딩보다 훨씬 나은 처리량
- 여러 요청 추론 교차 처리
- [LLM](/llm-kor.html) 효율성을 위해 필수

## 지연 시간 최적화

### 요청 배칭

오버헤드 상각을 위한 요청 그룹화:
- 단일 요청 지연 시간 증가 (배치 대기)
- 전체 처리량 극적으로 증가
- 지연 시간과 처리량 사이의 절충

### 모델 컴파일

모델을 대상 하드웨어로 컴파일:
- [양자화](/quantization-kor.html)를 통한 모델 크기 감소
- 특정 하드웨어에 대한 작업 최적화
- TensorRT (NVIDIA), CoreML (Apple), TFLite [양자화](/quantization-kor.html)

### KV 캐시 최적화

자동 회귀 모델 (언어 모델, [트랜스포머](/llm-kor.html)):
- 키-값 쌍 사전 계산 및 캐시
- 각 토큰에서 재계산 방지
- 메모리 대역폭 요구사항 감소
- 낮은 지연 시간 추론에 중요

### 혼합 정밀도 추론

FP32 대신 FP16/BF16 사용:
- 최신 하드웨어에서 2배 빠름
- 많은 모델에서 최소 정확도 손실
- A100, H100, 최신 GPU에서 기본 지원

## API 설계

### RESTful 추론

HTTP 기반, 간단한 통합:

```
POST /v1/predict
{
  "instances": [
    {"feature1": 1.0, "feature2": 2.0},
    {"feature1": 3.0, "feature2": 4.0}
  ]
}

응답:
{
  "predictions": [0.95, 0.12]
}
```

**장점**: 언어 무관, 쉬운 디버깅
**단점**: 높은 지연 시간, HTTP 오버헤드

### gRPC

바이너리 프로토콜, 고성능 RPC:

```protobuf
service PredictionService {
  rpc Predict(PredictRequest) returns (PredictResponse);
}
```

**장점**: 낮은 지연 시간, 이진 효율성, HTTP/2 멀티플렉싱
**단점**: 더 복잡, gRPC 클라이언트 필요

### 실시간 스트리밍

연속 예측용:
- 이벤트 스트림 (Kafka, Kinesis)
- WebSocket 연결
- 모니터링, 이상 감지에 유용

## 서빙 아키텍처 패턴

### 단일 모델 서빙

하나의 모델 버전 배포:
- 간단함, 낮은 오버헤드
- 버전 관리 복잡도 없음
- A/B 테스트 제한

### 다중 모델 서빙

여러 모델 버전 이용 가능:
- A/B 테스트: 다른 버전으로 트래픽 라우팅
- 모델 [앙상블](/ensemble-methods-kor.html): 예측 결합
- 신중한 자원 할당 필요

### 모델 앙상블

여러 모델 예측 결합:

```
predictions = [
  model_1.predict(x),
  model_2.predict(x),
  model_3.predict(x)
]
ensemble_output = average(predictions)
```

**이점**: 향상된 견고성, 더 나은 일반화
**비용**: 높은 지연 시간, 더 많은 자원 사용

## 배포 전략

### Blue-Green 배포

두 개의 동일한 환경:
- Blue (이전 모델), Green (새로운 모델)
- 모든 트래픽을 하나로 라우팅, 다른 하나 테스트
- 트래픽 전환으로 즉시 롤백

### 카나리 배포

점진적 트래픽 이동:
- 5% 트래픽을 새 모델로 라우팅
- 메트릭 모니터링
- 문제 없으면 증가: 10%, 25%, 50%, 100%
- 오류 감지시 자동 롤백

### 섀도우 배포

새 모델이 병렬로 실행:
- 프로덕션 모델과 동일한 요청 수신
- 예측 사용 안 함 (로깅만)
- 오프라인에서 예측 비교
- 사용자에게 위험 없음

## 모니터링 및 가시성

**메트릭**:
- 처리량 (요청/초)
- 지연 시간 (p50, p99)
- 오류율
- 하드웨어 사용률 ([GPU](/gpu-hardware-kor.html), 메모리, CPU)
- 모델 성능 (라이브 데이터의 정확도, 정밀도)

**경고**:
- 지연 시간 저하
- 오류율 급증
- 모델 성능 저하
- 자원 포화

**로깅**:
- 요청/응답 쌍 (디버깅용)
- 메타데이터가 포함된 모델 예측
- 오류 및 예외
- 오프라인 분석 및 모델 개선 활성화

## 비용 최적화

- **적절한 크기 조정**: 워크로드에 맞는 하드웨어 (과다 공급 금지)
- **스팟 인스턴스**: 중요하지 않은 서비스를 위해 저렴한 선점 가능 VM 사용
- **자동 확장**: 낮은 트래픽 기간 동안 확장 축소
- **모델 증류**: 더 큰 모델과 일치하는 더 작은 모델 교육
- **양자화**: 빠른 추론을 위한 정밀도 감소
- **캐싱**: 빈번한 쿼리 캐싱 (동일 입력 = 동일 출력)

## 결론

모델 서빙은 연구와 프로덕션 사이의 간격을 메웁니다. 성공은 지연 시간-처리량 절충 이해, 효율적인 아키텍처 설계, 포괄적인 모니터링 구현을 필요로 합니다. 최고의 모델도 신뢰할 수 있고 효율적으로 사용자에게 도달하지 못하면 무용지물입니다.
