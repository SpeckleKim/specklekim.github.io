# 벡터 데이터베이스

## 소개

벡터 데이터베이스는 고차원 벡터(임베딩)를 효율적으로 저장하고 검색합니다. 현대 AI 애플리케이션의 중심인 의미론적 검색, 유사성 매칭, [검색 증강 생성](/rag-kor.html)([RAG](/rag-kor.html)) 시스템을 지원합니다. 전통 데이터베이스는 정확한 일치를 인덱싱합니다. 벡터 데이터베이스는 유사성으로 검색합니다.

## 왜 벡터 데이터베이스인가?

### 유사성 검색 문제

쿼리 벡터 q와 수백만 개 벡터 모음이 주어졌을 때, k개 가장 유사한 것을 찾습니다:

**단순 접근**: q에서 모든 벡터까지 거리 계산 (O(n) 복잡도). 큰 데이터셋은 느림.

**인덱싱 접근**: 빠른 대략적 최근접 이웃(ANN) 검색을 가능하게 하는 구조 구축.

### 사용 사례

- **의미론적 검색**: 키워드뿐 아니라 의미 유사 문서 찾기
- **추천 시스템**: 임베딩 유사도 기반 아이템 추천
- **중복 감지**: 데이터셋 간 유사 콘텐츠 식별
- **RAG 시스템**: [LLM](/llm-kor.html) 추론을 위한 관련 컨텍스트 검색
- **이미지 검색**: 임베딩으로 시각적 유사 이미지 찾기

## 대략적 최근접 이웃(ANN) 알고리즘

### HNSW (계층적 네비게이션 가능 소세계)

계층적 그래프 구조 생성:
- 근접 그래프의 여러 계층 구축
- 상위 계층은 적은 수의 먼 노드 (네비게이션)
- 하위 계층은 조밀하게 연결 (정확도)
- 상위에서 하위로 네비게이션, 그 다음 로컬 검색

**장점**: 빠른 검색 (로그 복잡도), 좋은 품질

**절충**: 그래프 구조의 메모리 오버헤드, 느린 인덱싱

### IVF (역파일 인덱스)

벡터 공간을 영역으로 분할:
- 벡터를 k개 중심점으로 클러스터 (예: k-means)
- 각 벡터를 가장 가까운 중심점에 할당 (역 인덱스)
- 쿼리: 가장 가까운 중심점 찾기, 그들의 벡터만 검색

**장점**: 메모리 효율적, 병렬화 가능

**절충**: 품질은 중심점 선택에 의존, 빈 영역 가능성

### LSH (위치 민감 해싱)

유사 벡터를 같은 버킷으로 해시:
- 근처성을 보존하는 해시 함수 family 사용
- 유사 벡터는 높은 확률로 같은 버킷으로 해시
- 쿼리: 쿼리 벡터 해시, 같은 버킷 검색

**장점**: 이론적 보장, 구현 간단

**절충**: 높은 재현율을 위해 많은 해시 함수 필요

## 인기있는 벡터 데이터베이스 플랫폼

### Pinecone

완전 관리형 벡터 데이터베이스:
- 간편한 API, 유지보수 없음
- 자동 인덱싱 및 확장
- [LLM](/llm-kor.html) 생태계와 통합 (LangChain, LlamaIndex)
- 가격: 저장소 및 API 호출 기준

**적합**: DevOps 없이 프로덕션 수준 벡터 검색 원하는 팀

### Weaviate

오픈 소스, 유연함:
- 스키마 무료, GraphQL 인터페이스
- 내장 ML 모듈 (text2vec, image2vec)
- 하이브리드 검색 (벡터 및 키워드 결합)
- 클라우드 및 자체 호스팅 옵션

**적합**: 맞춤 요구사항, 온프레미스 배포

### Milvus

오픈 소스, 고성능:
- C++로 작성, 속도 최적화
- 여러 인덱스 유형 지원
- 분산, 클러스터 준비
- 활동적인 오픈 소스 커뮤니티

**적합**: 대규모 배포, 비용 의식 팀

### Chroma

경량, 개발자 친화적:
- 인프로세스 임베딩 저장소
- 외부 종속성 없음
- 프로토타이핑 및 로컬 개발에 우수
- 적절한 배포로 클라우드 확장 가능

**적합**: 빠른 프로토타이핑, 소규모-중규모 프로젝트

### Qdrant

기능 풍부한 오픈 소스:
- Rust로 구축, 고성능
- 고급 필터링, 페이로드 저장소
- 좋은 추천 시스템 지원
- 프로덕션 준비

**적합**: 복잡한 쿼리가 있는 프로덕션 시스템

### pgvector

PostgreSQL 확장:
- PostgreSQL에 벡터 검색 도입
- SQL 인터페이스, 기존 데이터베이스 활용
- 대략적 및 정확한 검색
- 제한적이지만 개선 중인 기능

**적합**: PostgreSQL 인프라가 있는 팀

## 인덱싱 전략

### 조밀 인덱스

모든 벡터를 빠른 액세스 메모리에 유지:
- 가장 낮은 지연 시간
- 가장 높은 메모리 비용
- 소규모-중규모 데이셋에 최적 (<1B 벡터)

### 희소 인덱스

데이터 분할, 필요할 때 로드:
- 분할 당 낮은 메모리
- 약간 높은 지연 시간
- 대규모 데이셋에 더 나음

### 계층적 인덱스

다양한 스케일에 최적화된 다단계 구조:
- HNSW는 자연스럽게 사용
- 둘의 장점: 속도와 메모리 효율

## 거리 메트릭

### 유클리드 거리

L2 노름: sqrt(sum((a_i - b_i)^2))
- 직관적, 일반적으로 사용
- 연속 특성에 적절

### 코사인 유사도

(a · b) / (||a|| ||b||)
- 범위 -1 ~ 1 (-1 = 반대 방향, 1 = 동일 방향)
- 텍스트 임베딩의 표준 ([정규화](/regularization-kor.html)됨)
- 스케일 불변

### 내적

a · b
- 빠른 계산 (단일 내적)
- 유사도 해석을 위해 [정규화](/regularization-kor.html) 벡터 필요
- 딥러닝에서 일반적

## RAG와의 통합

벡터 데이터베이스는 RAG의 핵심:

```
1. 문서 임베딩: 문서를 벡터로 변환, DB에 저장
2. 쿼리 처리: 사용자 쿼리 임베딩
3. 검색: k개 가장 유사한 문서 벡터 찾기
4. 컨텍스트 조립: 유사 문서의 전체 텍스트 검색
5. LLM 추론: 컨텍스트로 문서를 LLM에 전달
```

검색 품질은 최종 답변 품질에 극적인 영향을 미칩니다. 임베딩 모델 선택과 데이터베이스 설계가 중요합니다.

## 성능 고려사항

**처리량**: 초당 쿼리 수
- 인덱스 유형, 데이터 분포에 따라 다름
- HNSW 일반적으로 단일 코어 100-1000 QPS

**지연 시간**: 쿼리당 시간
- 최적화된 단일 쿼리는 밀리초 단위
- 배치 쿼리는 오버헤드 상각

**메모리**: 벡터당 저장소
- 압축 인덱스: 백만 벡터당 1-10GB
- 압축 안 함: 768차원 벡터 백만 당 4-8GB

**재현율**: 발견된 진정한 최근접 이웃의 백분율
- 재현율과 지연 시간 간 절충
- 높은 재현율 (>90%)은 일반적으로 10배 더 큰 검색 반경 필요

## 모범 사례

- **임베딩 모델 선택**: 기본, 도메인 특화 모델이 일반적으로 성능 우수
- **차원 축소**: PCA 또는 [양자화](/quantization-kor.html)로 메모리 감소, 최소 품질 손실
- **배치 작업**: 개별 쿼리보다 효율적
- **모니터링**: 쿼리 지연, 재현율, 인덱스 신선도 추적
- **정기적 재인덱싱**: 데이터 변경에 따라 인덱스 품질 저하 가능
- **거리 메트릭 일치**: 임베딩 모델에 적절한 메트릭 선택

## 미래 방향

- **멀티모달 벡터**: 텍스트, 이미지, 오디오를 위한 단일 임베딩 공간
- **학습된 인덱스**: 손작성 구조를 대체하는 ML 기반 인덱스
- **양자 접근**: 특정 유사성 검색을 위한 양자 컴퓨팅
- **하이브리드 조밀-희소**: 유연성을 위해 조밀 벡터와 희소 특성 결합

## 결론

벡터 데이터베이스는 AI 시스템에서 의미론적 이해를 가능하게 합니다. 올바른 데이터베이스 선택은 규모, 지연 시간 요구사항, 인프라 제약에 따라 달라집니다. RAG가 [LLM](/llm-kor.html) 애플리케이션의 표준이 되면서 벡터 데이터베이스 설계 이해는 AI 엔지니어에게 점점 더 중요해집니다.
