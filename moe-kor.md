# 혼합 전문가 모델 (MoE)

## 조밀 모델에서 희소 모델로

기존 트랜스포머 모델은 **조밀합니다**: 모든 토큰이 모든 계층과 모든 매개변수를 통과합니다. 이는 추론 비용이 모델 크기에 직접 따른다는 의미입니다. 70B 매개변수 조밀 모델은 모든 토큰에 대해 모든 매개변수를 처리하므로 예측 가능하지만 높은 계산 비용이 듭니다.

**혼합 전문가(MoE)**는 다른 접근 방식을 취합니다. 조밀 계층 대신 MoE는 각 토큰에 대해 매개변수의 부분 집합만 활성화되는 희소 계층을 사용합니다. 이는 토큰당 계산 비용을 상대적으로 일정하게 유지하면서 매개변수 수를 극적으로 확장할 수 있게 합니다.

### 조밀 vs 희소 트레이드오프

**조밀 모델**:
- 예측 가능하고 안정적인 성능
- 추론에 높은 계산 비용
- 모든 토큰이 모든 용량 사용
- 최적화 및 배포가 더 쉬움

**희소 모델 (MoE)**:
- 토큰당 낮은 추론 비용
- 학습 중 더 높은 피크 메모리
- 로드 밸런싱 문제
- 더 나은 효율성 가능성

## 게이팅 메커니즘

MoE의 핵심은 **게이팅 네트워크**입니다. 각 토큰에 대해 게이팅 함수는 어떤 전문가가 이를 처리할지 결정합니다. 일반적으로 게이트 벡터를 출력하는 학습된 신경망입니다—보통 전문가 선택에 대한 소프트맥스 확률입니다.

### 게이팅 작동 방식

위치 i의 각 토큰 x에 대해:
1. **게이트 점수 계산**: `g(x) = softmax(W_gate * x)`
2. **상위 k 전문가 선택**: 게이트 점수에서 상위 k 전문가만 유지
3. **전문가 출력 계산**: 전문가 j가 `E_j(x)` 계산
4. **결과 결합**: 게이트 점수를 사용한 가중 합

토큰 x의 결과: `y = Σ g(x)_j * E_j(x)` (선택된 전문가에 대해 합계)

이 게이팅은 완전히 미분 가능하여 모델이 학습 중에 토큰을 라우팅할 전문가를 학습할 수 있습니다.

## 상위-k 라우팅

대부분의 실용적 MoE 시스템은 **상위-k 라우팅**을 사용합니다: 각 토큰에 대해 게이트 점수에서 상위 k 전문가를 선택하고, k는 보통 2-8입니다. 이 희소 라우팅이 효율성 향상의 핵심입니다.

### 상위-k의 장점
- **계산 절약**: 모든 E 전문가 대신 k 전문가만 계산
- **메모리 효율성**: 조밀한 동등물과 비교하여 감소된 메모리 발자국
- **전문화**: 전문가들이 다양한 패턴에 전문화할 수 있음

### 로드 밸런싱 문제

학습된 라우팅의 문제: 전문가는 불균형해질 수 있습니다. 토큰이 일관되게 몇 가지 "쉬운" 전문가로 라우팅되면 다른 전문가는 비활성화되어 용량을 낭비합니다. 이를 **전문가 붕괴** 또는 **로드 불균형**이라고 합니다.

해결책:
- **보조 손실**: 불균형한 전문가 활용에 대한 페널티 추가
- **로드 밸런싱 게이트**: 라우팅을 수정하여 균형 잡힌 전문가 사용 권장
- **전문가 드롭아웃**: 학습 중 확률적으로 전문가 드롭
- **고정 라우팅**: 토큰을 균형 잡힌 방식으로 전문가에 사전 할당

## 주목할 만한 MoE 아키텍처

### Mixtral 8x7B

Mistral의 Mixtral 모델은 실용적 MoE 효과를 보여줍니다. 상위-2 라우팅이 있는 8개의 전문가(각 7B 매개변수)가 있습니다. 총 46.7B 매개변수에도 불구하고 토큰당 12.9B만 활성화합니다—34B 조밀 모델보다 활성 매개변수가 4배 적으면서 경쟁력 있는 성능을 가능하게 합니다.

주요 특징:
- 신중한 보조 손실 조정을 통한 균형 잡힌 로드로 학습
- 낮은 추론 비용으로 많은 벤치마크에서 더 조밀한 모델을 능가
- MoE가 소비자 등급 추론에 준비될 수 있음을 보여줌

### Switch Transformer

Google의 Switch Transformer는 상위-1 라우팅을 사용하여 MoE를 단순화했습니다. 각 토큰은 정확히 하나의 전문가로 라우팅되어 희소성과 단순성을 최대화합니다. 더 간단하지만 경험적 결과는 상위-k 접근과 비교하여 경쟁력 있는 성능을 보여줍니다.

주요 발견:
- 상위-1 라우팅은 구현을 단순화하고 통신 오버헤드 감소
- 적절한 추론 비용 증가로 극적인 매개변수 확장 가능
- 매우 큰 모델(수백억 매개변수)에 잘 작동

### GShard

GShard는 TPU 클러스터에서 600B+ 매개변수 모델을 학습하여 대규모 MoE를 입증했습니다. 기여:
- **분산 MoE**: 장치 전체에 전문가를 분할하는 기법
- **올-투-올 통신**: 원거리 전문가에 효율적으로 토큰 라우팅
- **확장성 패턴**: 거대한 희소 모델을 학습하는 방법

GShard는 조밀 모델로 불가능한 문제 규모로 MoE가 확장됨을 보여주었습니다.

## 로드 밸런싱 심화

로드 밸런싱은 불균형한 전문가 사용이 성능을 직접 저하시키기 때문에 중요합니다.

### 보조 손실

표준 접근법은 정규화 항을 추가합니다:
```
L_aux = α * Σ (f_i * L_i)
```
여기서 `f_i`는 전문가 i로 라우팅된 토큰의 비율이고 `L_i`는 해당 전문가의 손실입니다. 이는 균형 잡힌 활용으로부터의 편차를 페널티합니다.

그러나 보조 손실은 최적화와 충돌할 수 있습니다—균형 잡힌 라우팅 강제는 자연스러운 전문화를 방지할 수 있습니다.

### 용량 계수

다른 접근법은 용량 제약을 사용합니다. 각 전문가는 최대 용량(예: 전문가당 평균 토큰의 1.25배)을 가집니다. 이를 초과하는 토큰은 드롭되거나(위험) 재라우팅됩니다.

### 전문가 드롭아웃

학습 중 확률적으로 전문가를 비활성화하여 로드 밸런싱을 권장합니다. 이는 특정 전문가에 의존하지 않고 강력한 표현을 개발하도록 남은 전문가를 강제합니다.

## 효율성 향상

MoE의 주요 이점은 **추론 중 계산 효율성**입니다.

### 매개변수 vs 계산

상위-2 라우팅 및 8명의 전문가를 가진 100B 매개변수 MoE 모델은 토큰당 25B 매개변수만 활성화할 수 있습니다. 이것이 핵심 구별입니다:
- **매개변수**: 100B (모델 저장 메모리 요구사항)
- **활성 계산**: 25B (토큰당 계산)

40B 조밀 모델과 비교:
- **매개변수**: 40B
- **활성 계산**: 40B (토큰당)

MoE 모델은 매개변수 확장이 토큰당 비용을 직접 확장하지 않기 때문에 더 많은 유연성을 가집니다.

### 경험적 결과

연구는 다음을 보여줍니다:
- 10-100배 매개변수 증가(2-5배 계산 증가만 포함)
- 고정 계산 예산에 대해 조밀한 모델보다 나은 스케일링 법칙
- 희소 활성화에도 불구하고 경쟁력 있는 품질

그러나 추론 프레임워크는 이러한 이점을 실현하도록 최적화되어야 합니다—단순한 구현은 라우팅 오버헤드와 불규칙한 메모리 접근 패턴으로 인해 실제로 더 느릴 수 있습니다.

## 도전과제 및 트레이드오프

### 학습 복잡성
- 신중한 보조 손실 조정 필요
- 로드 밸런싱은 학습 초기에 불안정할 수 있음
- 튜닝할 하이퍼매개변수가 더 많음

### 통신 오버헤드
- 토큰 라우팅을 위해 장치 간의 올-투-올 통신
- 클러스터에서 네트워크 대역폭을 포화시킬 수 있음
- 분산 학습을 더 복잡하게 함

### 메모리 요구사항
- 모든 전문가는 클러스터 메모리에 맞아야 함
- 희소 활성화에도 불구하고 더 많은 전체 클러스터 메모리 필요 가능
- 로드 밸런싱은 종종 장치 전체에 토큰 복제 필요

### 하드웨어 제한
- GPU는 희소 작업에 최적화되지 않음
- 라우팅 오버헤드는 이론적 속도 향상을 무효화할 수 있음
- 실제 이점을 위해 신중한 엔지니어링 필요

## MoE 사용 시기

**적합한 경우:**
- 매우 큰 모델 용량 요구사항
- 추론 처리량에 민감한 애플리케이션
- 다양한 입력 유형의 시나리오(전문가 전문화)
- 예산 제약 배포(더 많은 용량/계산 달러)

**이상적하지 않은 경우:**
- 지연시간에 중요한 애플리케이션(라우팅이 오버헤드 추가)
- 단일 GPU 배포(라우팅이 비효율적)
- 메모리 제약 시나리오(모든 전문가 저장 필요)

## 미래 방향

활동적 연구는 다음을 포함합니다:
- **계층적 MoE**: 더 나은 확장을 위한 중첩된 전문가 선택
- **적응형 상위-k**: 토큰당 라우팅 예산 동적 조정
- **전문가 통합**: 전체 전문가 수 감소 훈련
- **개선된 로드 밸런싱**: 새로운 보조 손실 공식
- **하드웨어 최적화**: 희소 MoE 작업을 위한 CPU/GPU 지원

MoE는 더 효율적인 대규모 모델을 구축하기 위한 중요한 패러다임을 나타냅니다. 추론이 경제적으로 중요해짐에 따라 희소 모델은 LLM 배포에서 점점 더 중심적인 역할을 할 가능성이 높습니다.
